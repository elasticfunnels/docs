title: 'Imports'

Backend scripts support `import` / `export` syntax to share functions, constants, and objects across pages. Create a **backend script module** in your brand's Backend Scripts section, then import from it in any page's `<script scope="backend">` block.

## Quick example

Create a backend script with code `app` in your dashboard (Backend Scripts section):

```javascript
export const getGreeting = function (name) {
  return 'Hello, ' + name + '!';
};

export function isAdmin(email) {
  return email === 'admin@mybrand.com';
}
```

Import and use it in any page:

```html
<script scope="backend">
  import { getGreeting, isAdmin } from "app";

  var user = getCustomer();

  if (user && user.email) {
    setVariable('greeting', getGreeting(user.name));
    setVariable('is_admin', isAdmin(user.email));
    setVariable('user', user);
  }
</script>

<h1>{{ var.greeting }}</h1>

<!-- Debug: dump the full object -->
{{ dump(var.user) }}
```

<Warning>
`setVariable('name', value)` stores the value under the `var` namespace. Always use **`var.name`** when referencing the variable — e.g. `{{ var.user }}`, `{{ dump(var.user) }}`, `@if(var.user)`.

Writing `{{ user }}` or `{{ dump(user) }}` without the `var.` prefix will **not** find the variable.

Inside `@foreach` loops, the loop variable is its own scope and does **not** need `var.`:

```
@foreach(cat in var.categories)
  {{ cat.name }}          ← correct (loop variable, no var. prefix)
  {{ var.cat.name }}      ← wrong
@endforeach
```

See [Template Variables](/backend-scripts/actions#template-variables) for full details.
</Warning>

## What are backend script modules?

Backend script modules are standalone JavaScript files managed in your brand's **Backend Scripts** section. Each module has:

- **Code** — a slug-like identifier (e.g. `app`, `utils`, `auth-helpers`). This is what you use in `import` statements.
- **Content** — the JavaScript source code with `export` statements.
- **Status** — only `active` scripts can be imported.

Modules are brand-scoped — a page can only import scripts belonging to the same brand.

## Import specifier formats

All of these resolve to the same script code `app`:

```javascript
import { test } from "app";        // bare code
import { test } from "app.js";     // with .js extension
import { test } from "../app.js";  // relative path prefix
import { test } from "./app";      // relative path prefix
```

The resolution rules:
1. The `.js` extension is stripped
2. Leading `./` or `../` prefixes are stripped
3. The remaining string is matched against the script's `code` field

## Named imports

Import specific exports by name:

```javascript
// Backend script module (code: "utils")
export const TAX_RATE = 0.08;

export function calculateTax(amount) {
  return amount * TAX_RATE;
}

export function formatCurrency(val) {
  return '$' + val.toFixed(2);
}
```

```html
<!-- Page backend script -->
<script scope="backend">
  import { calculateTax, formatCurrency } from "utils";

  var price = 49.99;
  var tax = calculateTax(price);
  setVariable('total', formatCurrency(price + tax));
</script>
```

## Default imports

Use `export default` for a module's primary export:

```javascript
// Backend script module (code: "config")
export default {
  siteName: 'My Store',
  supportEmail: 'help@mystore.com',
  maxItems: 10
};
```

```html
<script scope="backend">
  import config from "config";

  setVariable('site_name', config.siteName);
  setVariable('support_email', config.supportEmail);
</script>
```

## Multiple modules

Import from several scripts in the same page:

```html
<script scope="backend">
  import { isAdmin } from "auth";
  import { getGreeting } from "helpers";
  import config from "config";

  var user = getCustomer();

  if (user && isAdmin(user.email)) {
    setVariable('greeting', getGreeting(user.name));
    setVariable('site', config.siteName);
  }
</script>
```

## Supported export syntax

| Syntax | Supported |
|--------|-----------|
| `export const name = ...` | Yes |
| `export let name = ...` | Yes |
| `export var name = ...` | Yes |
| `export function name() { ... }` | Yes |
| `export async function name() { ... }` | Yes |
| `export class Name { ... }` | Yes |
| `export default ...` | Yes |
| `export { name1, name2 }` | Yes |

## Limits

| Constraint | Value |
|-----------|-------|
| Max imports per script | 10 |
| Import depth | 1 level (nested imports inside modules are not resolved) |
| Script status | Only `active` scripts can be imported |

## How it works internally

When a page's backend script contains `import` statements:

1. Import statements are parsed and removed from the script
2. Each specifier is normalized to a script code (strip `.js`, `./`, `../`)
3. The `brand_backend_scripts` table is queried for an active script with that code
4. `export` keywords are stripped from the module code, turning exports into regular declarations
5. The module code is prepended to the page's script
6. The combined code runs as a single script in the QuickJS sandbox

This is a **compile-time inline** — not a runtime module loader. The imported code becomes part of the same function scope as the page's script, with full access to all sandbox globals (`setVariable`, `session`, `request`, `getCustomer`, etc.).
